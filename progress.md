# progress.md

## Project: Small-Model Tool-Use Reminder Harness

## Current Goal

Build backend + execution harness for small-model tool use in reminder domain.

## Status (2026-05-10)

Phase-1 implemented and verified:

- Python project skeleton created (`app/`, `tests/`, `scripts/`, `data/`).
- Reminder backend service implemented:
  - `create_reminder`
  - `query_reminder`
  - `update_reminder`
  - `delete_reminder`
- Local JSON storage implemented (`data/reminders.json`).
- Structured JSON statuses implemented across tool results:
  - `success`
  - `missing_fields`
  - `not_found`
  - `ambiguous`
  - `error`
- OpenAI-compatible tool schemas implemented in canonical registry (`app/tool_registry.py`).
- Tool executor implemented (`app/tool_executor.py`):
  - accepts `assistant.tool_calls`
  - parses JSON args
  - validates required fields via canonical schema
  - executes backend tool
  - returns `role=tool` messages
- Runtime loop implemented (`app/runtime_loop.py`) with max rounds to prevent infinite loops:
  - assistant emits tool call
  - runtime pauses and executes tool
  - runtime appends tool message
  - model is called again for final answer

## Update (chat web demo tool history rendering fix)

Fixed a likely 500 error in `scripts/chat_web_demo.py` after a successful first tool call. The backend keeps parsed tool-call arguments as JSON strings for `ToolExecutor`, but Qwen chat templates expect historical assistant `tool_calls.function.arguments` to be mappings when rendering the second model turn.

Added `messages_for_chat_template()` to build a render-only copy of messages that:

- removes private `_raw_preview` fields
- decodes JSON-string tool arguments into dicts for chat template rendering
- leaves the original runtime messages unchanged for backend execution

Also changed `/api/chat` to return JSON error payloads on exceptions, so the browser no longer fails with `Unexpected token 'I', "Internal S"... is not valid JSON`.
